# AI 模型流量控制系统 · 全架构落地方案（轻量转发 · 低延迟 · 全模型兼容）

## 一、核心架构总览（SaaS 化服务，毫秒级响应）

整体采用 **1 层核心转发网关 + 6 大能力中台 + 1 套客户端（控制台） + 1 套管理后台（客户管理端 + 模型管理端）** 架构，全程无业务逻辑耦合，仅做请求路由、鉴权替换、流量管控与计费扣费，不增加额外网络延迟，通过多协议原生透传兼容 OpenAI / Anthropic / Gemini 全品类 AI 模型。用户通过客户端（控制台）获取 API Key 并配置到各类调用端（OpenClaw / Cursor / Cline / Continue / Dify / OpenAI SDK 等），实际流量由调用端直接发往模型网关，系统内置防封号、流量最优分配、成本透明化与用户体验兜底机制。

**核心用户路径：** 用户注册登录 → 客户端（控制台）创建 API Key → 复制接入配置到调用端 → 调用端直接调用模型网关 → 客户端（控制台）查看用量 / 余额 / 日志

**核心链路：** 调用端请求 → 统一网关入口 → API Key 认证 → 余额 / 限流校验 → tokenGroup 路由匹配 → 协议适配 → IP 池调度 → 模型转发 / 缓存响应 → 实时计费扣费 → 监控上报

> **名词约定：** 本文档遵循项目统一名词体系，详见 [glossary.md](glossary.md)。「调用端」指通过 API Key 调用网关的服务（OpenClaw/Cursor/Cline 等）；「客户端（控制台）」指用户管理 Key 和余额的 Web 界面；「客户管理端」指管理用户充值与费用的后台模块；「模型管理端」指管理模型、路由、限流的后台模块。

### 高可用与水平扩展

- 网关层设计为**无状态服务**，所有有状态数据（限流计数、余额、IP 池状态）收敛至 Redis，网关实例可随时横向扩容。
- 前置 Nginx / Caddy 做负载均衡，最少部署 2 个网关实例，单实例故障时自动摘除，保障入口高可用。
- 支持平滑发布：新版本网关实例就绪后切流，旧实例排空连接后下线，升级过程零中断。
- **Redis 高可用：** 生产环境部署 Redis Sentinel 主从架构（或 Redis Cluster），避免 Redis 单点故障导致全站不可用；Redis 不可用时默认进入受限降级模式：暂停新请求或放行但停止扣费，避免在失去限流、余额和配额控制后出现超额消耗。
- **优雅停机：** 收到 SIGTERM 后立即停止接受新请求，等待存量流式请求完成（最长等待时间可配置，默认 120 秒），超时后强制断开并通过 `[STREAM_INTERRUPTED]` 标记通知调用端重试。
- **健康检查端点：** 网关暴露 `GET /healthz`（存活检查）和 `GET /readyz`（就绪检查，含 Redis + MySQL 连通性验证），供负载均衡器和 Docker 健康检查使用。
- **控制面不可用时的数据面降级：** 数据面本地缓存一份最近的路由 / 限流 / 模型配置快照（定期从控制面同步），控制面不可达时自动降级为只读模式，基于本地快照继续转发请求，避免控制面故障连带数据面不可用。

## 二、核心模块落地实现（全需求覆盖，可直接部署）

### （一）多协议原生透传层（固定入口 · 保留原生格式 · 对外暴露）

- 部署单一 HTTP / WebSocket 统一入口（如 `api.4tk.ai`），网关同时暴露多条协议路径，每条路径保持与官方 API 完全兼容的请求 / 响应格式，用户只需将调用端的 `base_url` 指向本网关、`api_key` 填入自己的 API Key 即可接入，无需任何额外适配。

#### 支持的协议路径

| 协议 | 路径 | 说明 |
|------|------|------|
| OpenAI Chat Completions | `POST /v1/chat/completions` | 兼容官方 Chat Completions，支持流式 `stream: true` |
| OpenAI Responses | `POST /v1/responses` | 兼容官方 Responses API，须为模型配置 `protocol=responses` 上游 |
| OpenAI Embeddings | `POST /v1/embeddings` | 兼容官方 Embeddings API，支持文本与多模态输入，仅非流式 |
| OpenAI Audio Speech | `POST /v1/audio/speech` | 兼容官方 Audio Speech API，返回音频二进制 |
| OpenAI Models 列表 | `GET /v1/models` | 根据 API Key 所属 tokenGroup 动态返回可用模型列表 |
| Anthropic Messages | `POST /v1/messages` | 兼容官方 Anthropic Messages API，鉴权使用 `x-api-key` + `anthropic-version` |
| Gemini generateContent | `POST /v1beta/models/{model}:generateContent` | 兼容官方 Gemini API，流式使用 `:streamGenerateContent`（带 `?alt=sse`） |

- 全链路非阻塞异步处理，支持高并发转发，端到端延迟 ＜10ms，不影响交互体验。
- 支持请求头 / 请求体动态改写，自动将用户 API Key 替换为对应上游的真实凭证。
- 支持 `reasoning_effort`（推理强度）参数的识别、透传和记录。
- **上游响应异常防御：** 网关对上游响应做最小化校验（HTTP 状态码 + Content-Type），当上游返回非预期格式（如 HTML 错误页、截断的 SSE 流、非 JSON 响应体）时，不直接透传给用户，而是走降级链路重试或返回标准化错误提示。

#### 入口认证机制

- 用户通过客户端（控制台）创建 API Key 后，将 Key 配置到各类调用端，调用端直接调用网关：
  - **OpenAI / Gemini 路径：** 请求通过 `Authorization: Bearer <API Key>` 传递，网关校验 Key 后替换为真实上游凭证。
  - **Anthropic 路径：** 请求通过 `x-api-key: <API Key>` 传递，同时建议携带 `anthropic-version: 2023-06-01`。
  - **HMAC 请求签名（可选增强）：** 对关键参数做 HMAC-SHA256 签名，防止 Key 被中间人截获后重放。
- 每个 API Key 绑定独立的 tokenGroup、限流配额和审计日志，支持随时吊销。

### （二）用户注册与登录体系（自助注册 · JWT 会话）

- **邮箱注册：** 用户通过邮箱 + 验证码自助注册，注册接口 `POST /auth/register/send-code`（发送验证码）、`POST /auth/register`（提交注册），注册成功后自动签发 JWT `accessToken`。
- **邮箱登录：** `POST /auth/login`，校验邮箱 + 密码，返回 `accessToken`；前端建议仅在内存中持有短期 `accessToken`，避免长期保存在 `localStorage` 中。
- **密码重置：** `POST /auth/reset-password/send-code`（发送重置验证码）、`POST /auth/reset-password`（提交新密码），全程自助，无需管理员介入。
- **邀请码机制（P2 可选）：** 每个注册用户可选绑定邀请码，用于后续激励、活动或外部推广场景；非核心链路，建议放在后续阶段实现。
- **会话管理：** 所有控制台 API 请求通过 `Authorization: Bearer <accessToken>` 鉴权，401 时前端自动清除 token 并跳转登录页。

#### 内部团队快速开通模式

- 若系统首期主要服务自有团队，建议优先支持邮箱注册 + 管理员开通 + 首次登录引导，让用户登录后即可进入控制台创建 Key 并开始使用。
- 用户默认独立存在，不引入部门、组织、子账户等层级；预算、模型范围、Key 规则均直接绑定到单个用户。

#### JWT 过期与续签

- `accessToken` 有效期 2 小时，签发时同时返回 `refreshToken`（有效期 7 天）。
- 前端在 `accessToken` 过期前或收到 401 响应时，自动使用 `refreshToken` 调用 `POST /auth/refresh` 静默续签，获取新的 `accessToken`，用户无感知。
- `refreshToken` 过期后需重新登录；`refreshToken` 支持单次使用后轮换（Rotation），防止被窃取后长期滥用。
- `refreshToken` 建议通过 `HttpOnly + Secure + SameSite` Cookie 下发与保存，不暴露给前端 JavaScript；管理后台可进一步采用纯 Cookie 会话，降低 XSS 导致的会话接管风险。

#### 认证端点防暴力破解

- **登录频率限制：** 同一邮箱连续 5 次登录失败后锁定 15 分钟；同一 IP 每小时登录失败上限 30 次。
- **验证码频率限制：** 同一邮箱 60 秒内只能请求一次验证码；同一 IP 每小时验证码请求上限 20 次。
- **兑换码防试探：** 同一用户每分钟最多尝试兑换 5 次；兑换失败统一返回"无效兑换码"，不区分"不存在"和"已使用"，防止枚举。

### （三）全模型统一认证适配器（OAuth + API Key · 快速注册）

- 标准化认证模型，统一封装 OAuth 2.0 授权、API Key 鉴权两套流程，兼容市面所有 AI 模型认证方式，自动处理 Token 刷新、凭证过期重试。
- 提供模型快速注册模板：仅需填写 3 项核心配置（模型厂商、认证类型、凭证信息），系统自动生成标准化接入配置，无需二次开发，1 分钟接入新模型。
- 凭证加密存储（AES-256），支持权限隔离，杜绝密钥泄露，适配前文流量窃取防御逻辑。

#### 凭证轮换流程

- 凭证到期前 N 天（可配置，默认 7 天）自动推送告警至管理员。
- 管理员确认后，系统自动生成新凭证并灰度验证（用新凭证发一次探活请求确认可用）。
- 验证通过后自动切流至新凭证，旧凭证保留 24 小时后删除，全程无需手动操作。

### （四）智能路由引擎（tokenGroup 分组 · 多协议匹配 · 精准分发）

- **模型 + 模型账号管理：** 每个模型可挂载多条模型账号（`model_accounts`），每条账号配置：上游地址、凭证、认证方式、协议标识（`protocol` 字段，取值 `chat` / `responses` / `messages` / `gemini` / `embeddings` / `speech` 等）、权重、启用状态。
- **tokenGroup 分组路由：** 管理员按令牌分组（tokenGroup）配置可用模型账号和模型范围，请求根据 API Key 所属分组匹配对应的模型账号。
- **双路由策略，零误分发：**
  - **强制指定路由：** 用户请求携带目标模型参数时，仅匹配标注支持该模型 + 该协议 + 该 tokenGroup 的下游线路，精准分发；
  - **智能兜底路由：** 未指定模型时，按余额剩余量、响应速度自动分配最优模型。
- **路由规则热更新：** 配置变更写入 MySQL 后通过 Redis Pub/Sub 广播变更事件，各网关实例收到后拉取最新配置；同时设置 30 秒兜底轮询，防止 Pub/Sub 消息丢失，无需重启服务即可实时生效。
- **差异化超时配置：** 支持按模型或按能力标签配置不同的上游请求超时时间（如 Embedding 请求 10 秒、普通对话 60 秒、Reasoning 模型 180 秒），避免统一超时导致长推理被截断或短请求等待过久。

#### 路由规则版本管理

- 所有路由 / 限流 / 缓存策略的变更实行**版本化存储**，每次修改自动生成新版本快照，记录变更人、变更时间、变更内容 diff。
- 支持**一键回滚**至任意历史版本，回滚操作秒级生效，防止误操作导致全量请求异常。
- 管理后台策略编辑页在保存前展示**变更影响预览**（本次变更影响哪些模型、哪些路由规则），降低误操作风险。

#### 四级优雅降级策略

故障发生时按以下层级逐级降级，每一层的切换条件和超时阈值均可在管理后台配置：

| 降级层级 | 策略 | 触发条件 |
|---------|------|---------|
| L1 | 同厂商同模型换 Key | 当前 Key 额度耗尽或被限流 |
| L2 | 同厂商降级模型 | 同厂商所有同级别模型 Key 不可用 |
| L3 | 跨厂商替代模型 | 同厂商全部模型不可用 |
| L4 | 返回友好错误 + 排队 | 所有可用模型均不可用 |

### （五）额度 & 限流风控引擎（防封号 · 实时管控）

- **全维度实时采集：** 定时调用各模型官方接口，自动拉取可用免费额度、会员剩余流量、当前调用频率、并发上限，数据同步延迟 ＜30s。
- **多级限流防护（防封号核心）：** 支持 QPS 限流、并发限流、单日流量上限、单 Token 额度上限，超出阈值自动熔断，切换备用模型。
- **上游 Key 级并发控制：** 多个用户可能共享同一个上游 Key，而上游厂商对单 Key 有并发限制。路由选择阶段引入"上游 Key 并发信号量"，单 Key 并发达到阈值时自动切换到同模型其他 Key，避免触发上游 429。
- **差异化策略配置：** 免费额度账号低频率宽松限流，付费会员账号高并发优先调度，避免免费额度滥用导致封号，贴合前文额度滥用防御逻辑。
- 支持额度预警：剩余流量低于阈值时，自动推送提醒，避免断流。

### （六）自有余额与计费体系（微美元精度 · 实时扣费）

#### 余额管理

- **微美元精度：** 账户余额采用微美元（1 USD = 1,000,000 microUSD）整数存储，避免浮点精度问题，支持极小额交易。
- **充值方式：**
  - **兑换码充值：** 用户在控制台输入兑换码（`POST /me/balance/redeem`），即时到账，结果同步写入余额流水。
  - **管理员手动充值：** 管理员在后台为指定用户充值，指定金额（USD），系统自动转换为微美元入账。
- **余额流水：** 每笔充值、扣费、调账均生成流水记录（`GET /me/balance/logs`），包含变动金额、变动前余额、变动后余额、变动类型（`reasonType`）。

#### 余额邮件提醒

- 用户可在控制台自助配置余额低阈值邮件提醒（`PATCH /me/balance-alert`）：开关、阈值（USD）。
- 当余额低于阈值时向注册邮箱发送一封提醒（每个「低于阈值」周期至多一封；充值回到阈值以上后可再次触发）。

#### 模型定价管理

- 管理员为每个模型配置计费模式和单价：
  - **按 Token 计费：** 配置输入单价 `inputUsdPer1M`（$/1M tokens）、输出单价 `outputUsdPer1M`（$/1M tokens）、推理 Token 单价 `reasoningUsdPer1M`（$/1M tokens，用于 o1/o3 reasoning tokens、Claude extended thinking 等，未配置时按 `outputUsdPer1M` 计费）。
  - **按次计费：** 配置单次价格 `perRequestUsd`（$/次）。
- 用户可在控制台查看模型定价列表（`GET /me/model-pricing`），按能力分组对比，点击模型名可复制。

#### 实时计费扣费

- **非流式请求：** 请求完成后，根据上游返回的 `prompt_tokens` / `completion_tokens`（或 Anthropic 的 `input_tokens` / `output_tokens`）按定价从余额实时扣费。
- **流式请求：** 先按预估上限预扣，流式完成后 SSE 完成事件中取实际 Token 数结算，多扣的差额退回。
- **流式请求中途断开处理：** 若调用端主动断开或流式超时（最大超时可配置，默认 5 分钟），网关强制结算已消耗部分的 Token，剩余预扣金额退回，避免预扣金额长期冻结。
- **缓存 Token 计费：** 区分缓存创建 Token（`cacheCreationTokens`）和缓存读取 Token（`cacheReadTokens`），按不同费率或折扣计费。
- **Audio Speech 计费：** 按 token 计费时按 prompt floor 与输出音频时长估算结算；按次计费时直接按单次价格扣费。
- 余额不足时拒绝请求，返回标准化提示。

#### 余额一致性保障

- 扣费采用 Redis 原子操作（`DECRBY`）实现高并发下的余额安全，扣费成功后异步写入 MySQL 流水记录。
- 定时对账任务（建议每 5 分钟一次）比对 Redis 余额与 MySQL 流水累计值，发现差异时自动生成调账记录并告警管理员。
- 极端情况下（Redis 宕机恢复后余额丢失），以 MySQL 流水为准重建 Redis 余额。

### （七）智能缓存引擎（省流量 · 重复请求极速响应）

#### P1：精确匹配缓存（首期实现）

- 对相同 prompt + 相同参数（model、temperature、max_tokens 等）的请求做精确哈希匹配，命中后直接返回缓存结果，不消耗模型 Token。
- 本地缓存 + Redis 二级缓存，响应延迟 ＜1ms，不影响交互流畅度。

#### P2：语义模糊缓存（数据验证后迭代）

- 采用 MinHash-LSH 模糊匹配算法，缓存高频重复请求、通用模板化问题，相似请求直接返回缓存结果。
- P1 上线后持续采集缓存命中率数据，当精确匹配命中率趋于稳定且低于预期时，再启动 P2 开发。

#### 通用缓存策略

- **缓存策略可配置：** 支持 TTL 过期时间、缓存白名单 / 黑名单，敏感问题强制跳过缓存，保障数据准确性。
- 缓存数据自动脱敏，不存储原始用户身份信息。

### （八）多 IP 出口调度系统（对接 Clash · 防 IP 封禁）

- 原生兼容 Clash 配置文件，自动解析多组代理节点、出口 IP 池，支持手动 / 自动两种 IP 切换模式。
- **智能调度策略：**
  - **轮询模式：** 每次请求切换不同出口 IP，分散请求来源；
  - **哈希模式：** 同一用户 / 同一模型绑定固定 IP，避免频繁切换导致风控；
  - **故障自动切换：** 节点超时 / 不可用，秒切备用 IP，无感知兜底。
- 支持 IP 黑白名单、地区筛选，规避高风险 IP 段，降低封号概率。

### （九）API Key 管理（用户自助 · 分组路由）

- **API Key 模型：** 每个注册用户可自行创建多个 API Key（sub-token），每个 Key 具有：
  - 名称（`name`）：用于标识用途，如「Cursor 开发」「Cline 生产」「测试脚本」。
  - 分组（`tokenGroup`）：绑定路由线路，须与管理员为该分组配置的线路一致方可正确路由，默认 `default`。
  - 状态（`isActive`）：支持启用 / 禁用 / 删除。
  - 最后使用时间（`lastUsedAt`）：自动记录。
- **明文安全：** API Key 明文仅在创建时展示一次供复制，服务端不存储明文，后续仅展示前缀掩码。
- **API Key 安全约束（建议 P1 支持）：**
  - 可选过期时间（如 30 天 / 90 天 / 不过期）。
  - 来源 IP 白名单（限定只能从指定 IP 调用）。
  - 可调用模型白名单（限定该 Key 只能访问指定模型）。
  - 单 Key 独立限流（RPM / TPM 上限）。
- **创建预设模板：** 创建 Key 时提供预设模板选项：「Cursor 开发用」「Cline 开发用」「生产环境」「测试脚本」「低成本批处理」，每个模板预设合理的分组和限流建议，用户确认即可创建。
- **管理接口：**

| 方法 | 路径 | 作用 |
|------|------|------|
| GET | `/me/tokens` | 列出当前用户所有 API Key |
| POST | `/me/tokens` | 创建 API Key，body：`{ name, tokenGroup, template?, expiresAt?, ipWhitelist?, modelWhitelist? }` |
| PATCH | `/me/tokens/:id/disable` | 禁用 API Key |
| PATCH | `/me/tokens/:id/enable` | 启用 API Key |
| DELETE | `/me/tokens/:id` | 删除 API Key |

### （十）可视化管理后台（全配置化 · 零代码操作）

#### 四层角色体系

> 详细名词定义见 [glossary.md](glossary.md)。

- **超级管理员（super_admin）：** 拥有最高权限，可访问**客户管理端**（用户充值、兑换码管理、充值记录）和**模型管理端**（模型接入、上游线路、令牌分组路由、限流规则、全局配置），以及创建普通管理员。
- **普通管理员（admin）：** 拥有日常运营权限，可访问**客户管理端**（用户充值、兑换码管理、充值记录），**不可访问**模型管理端。
- **普通用户（default）：** 通过**客户端（控制台）**管理 API Key、查看余额 / 日志、使用 Playground 做通路验证，无法访问任何管理端。

#### 模型管理端功能

> 以下功能仅 `super_admin` 可访问，在管理后台「模型管理」Tab 下操作。

#### 模型与上游线路管理

- **模型管理：** 新增 / 编辑 / 删除模型，每个模型配置：模型名、厂商、支持的能力标签、定价（per_token 或 per_request）、版本策略（跟随最新 / 锁定指定版本快照）。
  - **跟随最新：** 模型名直接映射上游最新版本（如 `gpt-4o` 始终指向 OpenAI 最新快照），上游更新自动生效。
  - **锁定版本：** 模型名映射到指定快照版本（如 `gpt-4o-2024-08-06`），上游发布新版本时不自动切换，需管理员确认后手动更新。
- **模型账号管理：** 每个模型可挂载多条模型账号（`model_accounts`），每条配置：上游地址、凭证、协议（`protocol`）、权重、启用状态，支持增删改。
- 删除模型时一并移除该模型的定价与模型账号配置。

#### 客户管理端功能

> 以下功能 `admin` 和 `super_admin` 均可访问，在管理后台「客户管理」Tab 下操作。

#### 运营管理

- **用户手动充值：** 管理员在后台为指定用户（按邮箱）充值指定金额（USD）。
- **兑换码管理：** 批量生成兑换码、查看兑换码列表（状态：已使用 / 未使用）、按邮箱 / 类型筛选。
- **充值记录：** 全站充值流水查询，支持按邮箱模糊搜索、按变动类型筛选。
- **管理员账户管理：** 超级管理员可创建 / 更新普通管理员账户。

#### 策略与系统管理

- **策略管理：** 可视化配置限流规则、额度阈值、路由策略、缓存规则，支持批量修改，保存前展示变更影响预览。
- **IP 池管理：** 导入 Clash 配置，管理代理节点，查看 IP 可用性、切换记录。
- **系统配置：** 统一入口管理、凭证加密、告警规则配置，全功能可视化操作。
- **操作审计：** 所有配置变更（策略修改、模型增删、凭证更换等）自动记录审计日志，包含操作人、操作时间、变更前后对比，日志不可篡改，保留 90 天。

#### 模型能力目录

- 管理后台维护一份**模型能力目录页**，展示当前系统已接入的所有模型及其：
  - 厂商、模型版本、支持的能力（对话 / 推理 / 嵌入 / 语音 / 多模态 / 图像生成等）
  - model name（可直接复制到调用端配置）
  - 推荐使用场景（如：代码生成推荐 Claude-3.5-Sonnet，通用对话推荐 GPT-4o）
  - 当前状态（可用 / 限流中 / 额度耗尽）、价格等级（低 / 中 / 高）
- 用户可在控制台浏览目录，按用途查看，如「通用对话」「代码辅助」「快速响应」「高质量推理」，降低非技术用户的选择成本。

### （十一）客户端 / 控制台（Key 管理 · 权益查看 · 日志排查）

#### 控制台首页

- **首页围绕"接入-可用-排障"三件事设计：**
  - 顶部：账户状态、余额 / 套餐、最近是否有失败调用。
  - 中间：API Key 列表，每个 Key 显示名称、状态、最后使用时间，提供一键复制 Key 和 `base_url` 按钮。
  - 下方：最近调用结果和失败原因摘要。
- **用户资料：** 展示邮箱、余额提醒状态。
- **快捷入口：** 创建 Key、查看接入文档、Playground 通路测试、查看推荐模型。

#### 接入向导（新用户首次登录引导）

- 新用户首次登录后，自动进入接入向导流程：
  1. **创建第一个 API Key：** 提供预设模板（Cursor / Cline / Continue / 通用），用户选择后一键创建。
  2. **复制接入配置：** 根据用户选择的工具，自动生成带 Key 和 base_url 的完整配置片段，一键复制。
  3. **发送测试请求：** 引导用户在 Playground 发一条固定测试消息，仅验证接入是否成功。
  4. **完成：** 跳转到控制台首页。
- 接入向导完成后不再自动弹出，用户可通过控制台"接入指南"入口随时重新查看。

#### 核心功能模块

- **API Key 管理：** 创建 / 启用 / 禁用 / 删除 API Key，Key 明文仅创建时展示一次并提供复制按钮（复制后显示绿色对勾反馈），后续仅展示前缀掩码。
- **推荐模型列表：** 以用途优先展示推荐模型，补充价格等级和适用场景；模型名支持一键复制。
- **调用日志：** 筛选最近调用记录（按 API Key / 成功或失败 / 模型关键字），展示字段：时间、Key 名称、模型、结果、耗时、消费 USD、失败原因、请求编号。默认展示"最近 1 小时"，支持"最近 1 小时 / 24 小时 / 7 天 / 自定义"快捷切换。
- **余额流水：** 查看充值、扣费和余额变化记录。
- **余额邮件提醒：** 自助开关和配置阈值。
- **兑换码充值：** 输入兑换码即时充值。
- **数据导出：** 调用日志和余额流水支持 CSV 导出。

#### Playground 测试面板

- 控制台内嵌轻量 Playground 页面，仅用于验证 Key、模型和线路是否打通，不承载日常使用流量。
- 核心交互：选择 API Key → 选择模型（下拉列表从 `/v1/models` 拉取）→ 输入一条测试语句 → 发送 → 返回成功 / 失败结果。
- 支持的协议模式：OpenAI 对话、OpenAI Responses、Anthropic Messages、Gemini 对话。
- 支持流式（SSE）和非流式（JSON）两种响应格式，但默认界面只展示是否打通和基础返回内容，不展示复杂技术指标。
- 提供几个预设测试 prompt 下拉选项（如"你好""返回一条测试回复""帮我确认接入是否成功"），方便快速验证。
- 保留最近 10 条测试记录在本地 `localStorage`，方便重复做通路检查。
- 模型调用走独立 API 域名（如 `api.4tk.ai`），若跨域请求失败自动回退到同源路径重试。

#### 控制台实时状态推送

- 仪表盘仅推送余额状态、最近失败请求和 Key 状态变化，无需手动刷新。
- **SSE 连接鉴权：** SSE 连接建立时校验 `accessToken`，连接最大存活时间与 `accessToken` 有效期一致（默认 2 小时），到期后服务端主动关闭连接，前端需重新鉴权后重连。
- **断线重连与事件补推：** SSE 事件携带递增 `eventId`，前端断线重连时通过 `Last-Event-ID` 头告知服务端断点，服务端从该点起重放缺失事件（最多保留最近 1000 条），避免断线期间数据丢失。

#### 国际化（i18n）

- 控制台和登录 / 注册页面支持中文 / 英文切换，语言偏好存储于 `localStorage`，全页面文案联动更新。

### （十二）API 接入文档（自动化 · 多工具适配）

- 提供独立的 `/docs` 接入文档页，包含：
  - **协议接入指南：** OpenAI Chat Completions、OpenAI Embeddings、OpenAI Audio Speech、OpenAI Responses、Anthropic Messages、Gemini generateContent 的请求示例和鉴权说明。
  - **当前可用模型列表：** 从模型定价接口自动拉取，展示每个模型支持的协议端点和定价。
  - **IDE / 工具配置模板：** Cursor、GitHub Copilot、Codex、Continue、Cline / Roo、Windsurf 等编码工具的完整配置说明，带 base_url 和 API Key 占位符。
  - **编排平台接入：** Dify、n8n 等自动化平台的供应商配置方式。
  - **快速验证命令生成器：** 用户选择协议 + 填入 API Key 后，自动生成可直接执行的 `curl` 命令，一键复制到终端验证接入是否成功。
  - **常见问题（FAQ）：** 401 鉴权失败、模型不存在、协议不匹配、余额不足等常见问题的排查指引。
- 文档页内置 API Key 选择器，登录用户可自动将 Key 填充到代码示例和 curl 命令中。

### （十三）全链路监控系统（多维度数据 · 实时观测）

核心监控指标，大屏实时展示：

- **厂商维度：** 接入模型厂商数量、各厂商支持模型类型；
- **流量维度：** 单模型接口调用量、实时流量带宽、累计流量消耗；
- **限流维度：** 当前限流阈值、已触发限流次数、限流状态；
- **余额维度：** 全站总余额、单用户余额分布；上游模型额度：单模型总可用流量 / 剩余流量，无限制时展示「不限」；
- **异常监控：** IP 封禁、额度耗尽、接口超时、限流触发实时告警，支持邮件 / 钉钉推送；
- **用户维度：** 单用户用量趋势、模型偏好分布、各 Key 用量分布、异常调用检测。

数据留存：7 天明细数据、30 天汇总数据，支持导出报表。

#### 请求链路追踪

- 每个请求分配唯一 `X-Request-Id`，通过响应头返回给调用方。
- 管理后台支持按 Request ID 查询完整调用链路：命中的路由规则 → 选择的模型 / Key → 出口 IP → 是否命中缓存 → 上游响应耗时 → 最终状态码。
- 团队成员反馈问题时，提供 Request ID 即可快速定位全链路，无需逐层排查。

### （十四）成本可视化看板（精准统计 · 成本透明）

#### P1：成本可视化看板（首期实现）

- **精准 Token 统计：** 兼容 OpenAI / Anthropic / Gemini 等全厂商 Token 计算规则，实时统计请求 Token、响应 Token、缓存创建 Token、缓存读取 Token、总消耗 Token，流式传输逐帧计量无误差。
- **团队成本总览：** 实时展示当月总花费、各模型成本占比、日/周成本趋势曲线、环比变化率。
- **成员用量排行：** 各成员 Token 消耗排行、模型使用分布、各 API Key 消耗排行，回答"谁用了最多 Token""哪个 Key 花费最高"的问题。
- **成本预警：** 当月成本超过预设阈值时自动告警，避免无感知超支。

#### P2：多维度计费（按需迭代）

- 支持按模型类型、账号类型（免费 / 会员）设置差异化单价，自动生成日 / 周 / 月计费账单。
- 适用于团队对外提供 API 服务或需要内部结算的场景。

### （十五）极致用户体验优化（兜底保障 · 无感知交互）

- **低延迟优化：** 全链路异步非阻塞，缓存优先响应，流式传输无卡顿，对话体验与官方直连一致；
- **故障无感知兜底：** 模型超时、IP 封禁、额度耗尽时，按四级降级策略自动切换备用模型 / IP，用户侧无中断、无报错；
- **兼容全终端：** 支持网页端、调用端（IDE / OpenClaw）、API 调用、IDE 插件，各协议保持原生格式，用户只需改 `base_url` + `api_key` 即可接入，无适配成本；
- **错误友好处理：** 拦截官方报错，返回标准化友好提示并附带操作引导（如："当前模型额度已用尽，已自动切换至 GPT-4o-mini，如需使用原模型请联系管理员补充额度"），不暴露底层转发、IP 切换细节；
- **路由透明度：** 响应头包含 `X-Actual-Model`、`X-Route-Info` 等技术细节，供调用端开发者排障使用；模型管理端可查看完整链路信息；
- **优先级调度：** 高频用户、关键请求优先分配优质 IP + 高额度模型，保障核心体验。

#### 流式传输故障处理

- 流式传输过程中若上游模型中断（超时 / 断连 / 报错），网关立即终止当前流并在流中插入特定标记 `[STREAM_INTERRUPTED]`。
- 调用端检测到中断标记后，自动携带原始请求重试（走降级链路选择备用模型），对用户展示"正在重新生成…"提示，而非直接报错。
- 若中断发生在输出前 20% 以内，直接静默重试；若已输出较多内容，保留已输出部分并提示用户"回答中断，正在重新生成剩余部分"。

#### 请求排队与等待反馈

- 当所有可用模型均处于限流或过载状态时，请求进入排队队列（队列上限可配置，默认 100）。
- 入队后立即返回预估等待时间（基于当前队列长度和平均处理速度计算），调用端展示等待进度。
- 队列满时返回标准化提示："当前请求量较大，请稍后重试"，附带建议重试时间。

## 三、核心技术选型（轻量易部署 · 低运维成本）

| 类别 | 选型 |
|------|------|
| 转发网关 | Go 开发（高性能低延迟），仅 3MB 轻量化部署 |
| 客户端（控制台） & 管理后台（客户管理端 + 模型管理端） | Go（Gin）提供 API + 前端 SPA（原生 HTML/CSS/JS + Bootstrap 5）内嵌至同一二进制，单文件部署 |
| 配置存储 | MySQL（用户 / 余额 / 模型 / 策略配置 / 审计日志 / 兑换码）+ Redis Sentinel 主从（缓存 / 限流 / IP 池 / 队列） |
| 数据库迁移 | golang-migrate / goose，所有 DDL 变更走 migration 文件，部署时自动执行 |
| 监控体系 | Prometheus + Grafana，开箱即用可视化大屏 |
| IP 调度 | 对接 Clash REST API，自动解析节点配置，无需二次开发 |
| 邮件服务 | SMTP 发送验证码和余额提醒邮件 |
| TLS | 生产环境强制 HTTPS（TLS 1.2+），HTTP 请求 301 跳转 HTTPS |
| 部署方式 | Docker 一键部署，支持单机 / 轻量服务器，无复杂依赖，最少 2 实例保障高可用 |

### 部署拓扑建议（国内控制面 + 海外数据面）

- 若主要上游为 OpenAI / Anthropic / Gemini，建议**模型转发网关、出口代理池、上游探活任务**优先部署在海外节点（推荐香港 / 新加坡 / 日本），以保证访问上游模型的稳定性和时延可控。
- **控制台前端、管理后台、接入文档、登录注册、报表与审计查询**可优先部署在国内，保障团队成员日常访问体验。
- 调用端（OpenClaw / Cursor / Cline 等）通过 `api.4tk.ai` 域名访问海外网关节点；客户端（控制台）通过 `console.4tk.ai` 域名访问国内服务。
- Clash / 代理能力仅作为服务端出口治理能力使用，不作为终端用户侧的使用前提。
- 若用户、审计、财务等数据存在较强本地合规要求，可将 MySQL 主库放国内；海外网关通过专线、白名单或只读副本访问必要数据，避免让全部业务数据随数据面一起出海。
- **跨地域配置同步：** 控制面（国内）配置变更后通过 Redis Pub/Sub 广播至海外数据面，最大同步延迟 ≤5 秒；同时提供管理后台"强制刷新"按钮，供紧急变更场景下手动触发海外节点立即拉取最新配置。

## 五、防风险兜底机制（贴合流量防御逻辑）

- **传输安全：** 生产环境强制 HTTPS（TLS 1.2+），HTTP 请求 301 跳转 HTTPS，所有携带 API Key 和凭证的请求均通过加密通道传输。
- **凭证安全：** 密钥 AES-256 加密存储，加密密钥通过环境变量注入、启动时内存解密，不落盘存储；后续可对接 KMS（云上用云 KMS，自部署用 HashiCorp Vault）；权限最小化，定期自动轮换高危凭证（见凭证轮换流程）。
- **用户密码安全：** 用户登录密码 bcrypt 哈希存储，不可逆；API Key 明文仅创建时返回一次，服务端仅存储哈希。
- **API Key 安全加固：** 支持过期时间、来源 IP 白名单、可调用模型白名单、单 Key 独立限流，降低 Key 泄露后的损害范围（见 API Key 管理）。
- **认证防护：** 登录、验证码、兑换码等端点实施频率限制，防止暴力破解和枚举攻击（见认证端点防暴力破解）。
- **CORS 策略：** 控制台域名（如 `console.4tk.ai`）仅允许受信前端 Origin 访问控制台 API；模型网关域名（如 `api.4tk.ai`）按需配置 CORS（IDE 插件等非浏览器场景不受限）。
- **流量防护：** 拦截异常批量请求、恶意高频调用，避免账号被牵连风控。
- **请求内容审计：** 对请求内容做在线风险检测，但不落库存储原始 prompt；日志仅保留不可逆摘要（如哈希）、模型、Token 数、风险标签与告警结果，避免通过日志反推原始请求内容。
- **数据隔离：** 用户请求数据不持久化，缓存数据自动脱敏，杜绝数据泄露。
- **操作审计：** 管理后台所有配置变更记录不可篡改的审计日志，包含操作人、时间、变更 diff，保留 90 天。
- **上游 Key 泄露检测：** 定时探活任务除检测"能否调通"外，同时监控上游余额突降、异常 4xx 响应率飙升等信号；发现疑似泄露的 Key 自动禁用并告警管理员，防止被盗用产生高额消耗。
- **紧急冻结：** 支持管理员一键冻结某用户全部 API Key 或某团队全部访问权限，冻结操作秒级生效并自动触发告警通知相关成员，用于异常访问后的快速止血。
- **数据库备份：** MySQL 至少每日一次全量备份 + binlog 实时备份，备份文件异地存储；定期（建议每月一次）执行恢复演练，确保备份可用。
- **合规适配：** 内置请求频率合规控制，规避跨境流量风控，符合使用规范。

## 六、面向团队自用场景的优化建议（建议纳入 P1 / P2 路线图）

当前方案已具备完整 SaaS 化能力；若本系统的首要目标是**服务自有团队、控制全部 AI 流量、降低使用成本、提升治理能力**，建议在实施时优先收敛为"团队流量治理平台"，按以下方向优化。

### （一）架构优化建议（先治理、后扩张）

- **P1 先聚焦核心链路：** 首期仅实现统一网关、API Key 鉴权、限流、预算 / 配额、基础路由、调用日志、监控告警，不急于上线邀请码、兑换码、返利、复杂语义缓存等偏运营能力，先把稳定性和治理能力做扎实。
- **控制面 / 数据面彻底拆分：**
  - **数据面：** 只负责请求接入、API Key 认证替换、限流、路由、转发、降级，不承载后台复杂业务逻辑；
  - **控制面：** 负责用户、Key 管理、模型配置、策略管理、审计日志和报表。
  - 拆分后可避免后台复杂逻辑拖慢主请求链路，也便于后期独立扩容。
- **路由保持后台托管：** 建议继续以 `tokenGroup` 为主，但路由、降级、熔断、成本优先级全部由后台维护，不把复杂策略暴露给普通用户。
- **限流采用三层结构：** 建议具备全局、用户、API Key 三层限流与并发控制，避免单个用户或单个 Key 占满全部资源。
- **预算与配额拆开治理：**
  - **预算：** 控制花多少钱（按日 / 周 / 月预算）；
  - **配额：** 控制可用多少次、多少 Token、多少并发。
  - 这样更适合单用户独立治理和后续精细化控制。
- **显式降级策略：** 降级不应仅按"可用即切"，而要按任务类型限制降级范围，例如代码生成优先降到同类高质量模型，Embedding 仅能切换到同能力模型，避免功能错配。
- **缓存策略先精确后模糊：** P1 仅做"完全相同参数 + 完全相同请求"的精确缓存；语义缓存延后，避免错误命中影响结果可信度。
- **策略版本化 + 灰度发布：** 除版本快照与回滚外，建议新增灰度发布能力，支持按用户或按比例流量先验证新规则，再逐步全量生效。
- **预留沙箱空间：** 新模型、新路由、新限流规则先在测试用户或测试 Key 中试运行，通过后再放量到全量用户。
- **统一链路标识：** 所有请求从 P1 起统一记录 `requestId`、`userId`、`tokenId`、`routeRuleId`、`model_account_id`，为后续排障、审计、结算、统计打基础。
- **客户端（控制台） / 管理后台接口统一版本前缀：** 建议所有客户端（控制台）和管理后台接口加 `/api/v1` 前缀（如 `/api/v1/auth/login`、`/api/v1/me/tokens`），为后续不兼容变更预留空间；模型网关接口保持与上游一致的版本路径不变。
- **余额一致性对账机制：** Redis 原子扣费 + 异步写 MySQL 流水，定时对账任务（每 5 分钟）比对两端数据，差异自动调账并告警；Redis 宕机恢复后以 MySQL 流水为准重建余额。

### （二）安全优化建议（把治理风险前置）

- **API Key 默认最小权限：** 除明文只展示一次、服务端仅存哈希外，建议支持过期时间、来源 IP 白名单、可调用模型白名单、调用时间窗等约束，降低 Key 泄露后的损害范围。
- **管理员必须启用双重认证：** 普通用户可保持邮箱 + 密码登录，管理员及高权限操作建议强制启用 2FA，并对修改策略、查看敏感日志、导出报表等操作做二次确认。
- **权限模型保持简单：** 先维持管理员 / 普通用户双角色模型；若后续后台管理人员增多，再逐步引入资源级权限控制。
- **高危操作强审计：** 创建 / 禁用 / 删除 Key、修改路由、调整限流阈值、切换上游凭证、导出数据等操作均应记录不可篡改审计。
- **请求签名建议默认用于内部接入：** 对 CLI、脚本、服务到服务调用，建议默认启用带时间戳、随机数、有效期的 HMAC 签名，防止重放攻击。
- **凭证管理尽早接入 KMS / Vault：** 当前环境变量 + 启动时解密可作为过渡方案，但正式落地建议对接 KMS 或 Vault，支持主密钥轮换、环境隔离、解密审计。
- **日志默认不可逆脱敏：** 不建议保留可推断原文的摘要信息，敏感字段统一哈希或分类标签化，确保即使日志泄露也无法还原用户请求。
- **错误信息分层输出：**
  - **用户侧：** 返回友好、可操作的错误提示；
  - **内部侧：** 记录完整上游错误、路由信息、代理信息，供排障使用；
  - 避免将真实上游结构、代理细节暴露给普通调用方。
- **引入异常行为检测：** 除限流外，增加对夜间爆量、跨地区突增、多 Key 共用同一出口、异常失败率飙升等行为的检测与告警。
- **出口代理池独立保护：** 代理节点配置、出口 IP 信息、切换策略应视为高敏资产，与业务配置分开授权，避免普通管理员直接接触底层网络资源。

### （三）用户视角优化建议（让用户更容易接入和使用）

- **模型目录按用途展示：** 按「通用对话」「代码辅助」「快速响应」「高质量推理」等用途展示，标注推荐模型，model name 支持一键复制到调用端配置。
- **首页不强调技术指标：** 用户侧优先展示账户状态、可用额度、最近是否有失败请求、推荐接入入口，不堆积 Token、并发、缓存命中率等专业指标。
- **拒绝原因可解释：** 当请求被拒绝、排队、降级时，API 响应体中应包含明确原因字段，如 `"error": "team_budget_exceeded"`、`"error": "model_not_allowed_for_this_key"`、`"error": "rate_limit_reached"`，而非仅返回通用 429/403。
- **Playground 只承担通路验证：** 不往复杂交互台演进，只需要让用户快速确认 `Key + 模型 + base_url` 是否可用。
- **Webhook / 异步通知：** 余额耗尽、Key 被禁用、模型下线、限流触发等事件除邮件通知外，建议支持 Webhook 回调，便于接入团队自己的飞书 / Slack / 钉钉机器人。
- **批量运营能力：** 除批量生成兑换码外，增加批量充值（给一批用户统一充值）、批量调整配额、批量启停 Key 等运营能力。
- **用量数据导出：** 用户调用日志和余额流水支持 CSV 导出，便于管理员做统计和核对。

### （四）用户体验优化建议（降低接入和排障成本）

- **控制台首页突出四个关键动作：** 创建 Key、复制接入配置、查看推荐模型、排查最近失败请求；避免一上来堆积大量复杂指标。
- **接入向导是新用户的第一体验：** 新用户首次登录后直接进入"创建 Key → 选择工具 → 复制配置 → 发送测试请求"引导流程，3 分钟内完成接入。
- **一键复制交互必须顺滑：** 控制台多处提供 base_url 和 API Key 的一键复制按钮，复制后显示绿色对勾反馈，这是最高频操作。
- **错误提示必须附带下一步动作：** 如"当前模型限流，已自动切换备用模型""余额不足，请联系管理员充值""当前 Key 仅允许访问指定模型"等，帮助用户自助解决问题。
- **日志页优先支持失败排查：** 建议默认展示最近失败请求、可按 `requestId` 和 Key 名称快速追踪命中规则和上游状态，而不只是按字段筛选。
- **流式失败重试策略可配置：** 根据系统稳定性策略允许配置静默重试、提示后重试、完全不自动重试三种模式。
- **排队反馈透明化：** 除返回预计等待时间外，建议同时展示"等待可用并发""正在尝试备用线路""已切换低成本模型"等状态，减少"系统卡住"的感知。
- **增加模型健康状态页：** 对外展示各模型当前状态（可用、限流、维护中、额度不足），让用户在配置调用端前就知道当前系统健康度。
- **管理后台增加策略回放验证：** 在策略保存前后，支持模拟"某用户 / 某 Key / 某模型请求"会命中哪条路由与限流规则，降低误操作风险。
- **默认即好用：** 新用户注册后，系统自动分配默认推荐模型、默认 Key 模板和默认限流策略，避免首次接入就面对大量空配置。
